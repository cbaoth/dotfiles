# sshd password auth stayed on: drop-in ordering is first-wins, not last-wins

**The hardening file was correct and had no effect.** `99-hardening.conf` said
`PasswordAuthentication no`; sshd accepted passwords anyway, for five days after
the rebuild. Nothing in the file was wrong — the *filename* was.

## Symptom

Interactive login on port 8090 succeeded with the user's Unix password, on a box
that was supposed to be key-only since the 2026-07-18 rebuild. Key auth also
worked, so nothing looked broken; the extra path was simply open.

Reading `/etc/ssh/sshd_config.d/99-hardening.conf` shows the intended config and
explains nothing. `sshd -t` reports the config valid. Both are true and both are
irrelevant.

## Cause

`sshd_config` puts `Include /etc/ssh/sshd_config.d/*.conf` at line 24, above
every other directive, and **OpenSSH uses the first obtained value for each
keyword**. Drop-ins are read in lexical order, so the *lowest*-numbered file
wins — the opposite of the systemd / `sysctl.d` / `sudoers.d` convention that
the `99-` prefix is borrowed from.

What was actually on disk:

| File | `PasswordAuthentication` | Effect |
| ---- | ------------------------ | ------ |
| `50-cloud-init.conf` | `yes` | **wins** |
| `60-cloudimg-settings.conf` | `no` | shadowed |
| `99-hardening.conf` | `no` | shadowed |

`50-cloud-init.conf` is generated by cloud-init's ssh module from
`/etc/cloud/cloud.cfg.d/99-ssh.cfg`, written by the Contabo reinstall:

```yaml
ssh_pwauth: 1
ssh_deletekeys: 0
```

So deleting the drop-in is not enough — cloud-init re-renders it. The source
setting has to go too.

Only `PasswordAuthentication` was affected. `Port 8090`, `PermitRootLogin no`,
`MaxAuthTries 3` and `AllowUsers` had no competing entry in a lower-numbered
file, so the rest of the hardening was genuinely in force the whole time.

## Fix

```bash
sudo sed -i 's/^ssh_pwauth:.*/ssh_pwauth: 0/' /etc/cloud/cloud.cfg.d/99-ssh.cfg
sudo rm /etc/ssh/sshd_config.d/50-cloud-init.conf
sudo mv /etc/ssh/sshd_config.d/99-hardening.conf \
        /etc/ssh/sshd_config.d/01-hardening.conf
sudo sshd -t && sudo sshd -T | grep -E '^(passwordauthentication|permitrootlogin|pubkeyauth|port|allowusers)'
sudo systemctl reload ssh
```

The `01-` rename is the durable half: it makes the hardening authoritative
against anything the image or cloud-init writes later, instead of depending on a
competing file staying deleted. Fixing `ssh_pwauth` alone would work today and
break again on the next reinstall.

Verified: password login now rejected, key login (agent) works, and
`sshd -T` reports `passwordauthentication no`.

## The lesson that generalises

**`sshd -t` validates syntax, not intent.** A file can be syntactically perfect,
semantically correct, and completely inert. The only honest check is
`sshd -T`, which prints the *effective* config after includes are resolved:

```bash
sudo sshd -T | grep -E '^(passwordauthentication|permitrootlogin|kbdinteractive)'
```

Any future "is this hardened?" question about sshd should be answered from
`sshd -T`, never from reading a drop-in.

## Also found while in there

- `/etc/ssh/sshd_config` was replaced by a package upgrade on 2026-07-23 14:30
  (ucf; now byte-identical to `sshd_config.ucf-dist`, sshd restarted 14:29:43).
  The shipped file actively sets **`PermitRootLogin yes`** at line 54 — not
  commented out. It is overridden by the drop-in, but it means root-by-password
  is one broken `Include` away. Keeping all hardening in drop-ins is what made
  this upgrade a non-event; do not start editing the main file.
- `99-hardening.conf.bak-20260721213355` (pre-`pull-backup` `AllowUsers`) was
  inert — the include glob is `*.conf` — but removed as clutter.

## Automation

`bootstrap-new-server.sh` (gitignored, `_local/vserver-migration/`) wrote the
`99-` name and therefore reproduced this bug on every rebuild. Fixed
2026-07-23 to:

1. write `01-hardening.conf`, with the first-wins rule stated in a comment,
2. set `ssh_pwauth: 0` and delete `50-cloud-init.conf` plus the legacy `99-` file,
3. **assert `sshd -T` output** for `passwordauthentication no`, `permitrootlogin
   no`, `kbdinteractiveauthentication no`, `pubkeyauthentication yes` and `die`
   before restarting if any is missing.

Step 3 is the one that matters: it turns a silent shadowing into a hard failure
during bootstrap, when there is still a root session open to fix it.

Not extracted into a `setup/` module — this is vserver bootstrap, which runs
before `dotfiles-link` and is not part of the `system-setup` lifecycle. See
`docs/setup/vserver-migration.md`.
